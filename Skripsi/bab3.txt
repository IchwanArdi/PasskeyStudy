BAB III
METODE PENELITIAN
A.	Tempat dan Waktu Penelitian
Penelitian ini dilaksanakan pada bulan Januari hingga Juni 2026, mencakup seluruh tahapan mulai dari analisis kebutuhan, perancangan, implementasi, hingga pengujian sistem. Observasi dan wawancara awal dilakukan langsung di Kelurahan Karangpucung Purwokerto Selatan pada bulan Januari 2026 sebagai dasar identifikasi kebutuhan sistem. Pengembangan dan pengujian dilakukan secara daring menggunakan perangkat pribadi peneliti dengan deployment pada platform Vercel sehingga sistem dapat diakses melalui internet tanpa memerlukan infrastruktur server fisik di kelurahan.
B.	Metode Pengumpulan Data
1.	Studi Literatur
Studi literatur dilakukan dengan mengumpulkan dan mengkaji referensi yang berkaitan dengan FIDO2/WebAuthn, autentikasi passwordless, keamanan sistem informasi, dan layanan administrasi kelurahan digital. Sumber referensi mencakup jurnal nasional, jurnal internasional, dan dokumentasi resmi standar WebAuthn dari W3C dan FIDO Alliance. D.	Konsep Penelitian
Penelitian ini menggunakan metode Research and Development (R&D) dengan model Prototyping sebagai kerangka pengembangan sistem. Metode R&D dipilih karena penelitian ini bertujuan menghasilkan produk nyata berupa prototipe sistem autentikasi passwordless berbasis FIDO2/WebAuthn pada website layanan administrasi Kelurahan Karangpucung (Rahayu, 2025). Model Prototyping dipilih karena implementasi WebAuthn melibatkan interaksi kompleks antara browser, perangkat autentikator, dan server yang memerlukan pengujian bertahap sehingga memungkinkan perbaikan sistem secara iteratif berdasarkan hasil evaluasi setiap siklus (Nurlelah dkk., 2023).
Pengembangan sistem dilakukan dalam tiga iterasi Prototyping dengan lingkup yang jelas dan terukur sebagaimana ditunjukkan pada Gambar 3.2.

b)	Perancangan Arsitektur Sistem
Sistem dirancang menggunakan arsitektur client-server dengan pemisahan antara frontend dan backend. Frontend dirancang menggunakan React.js dengan Vite dan bertanggung jawab menyediakan antarmuka pengguna serta mengelola interaksi WebAuthn melalui library @simplewebauthn/browser. Backend dirancang menggunakan Node.js dengan Express dan bertanggung jawab mengelola logika bisnis serta operasi kriptografi WebAuthn melalui library @simplewebauthn/server. MongoDB digunakan sebagai basis data untuk menyimpan data pengguna, kredensial WebAuthn, data pengajuan surat, dan recovery codes. Komponen Authenticator berupa sensor biometrik perangkat (sidik jari) atau security key berperan sebagai perangkat yang melakukan operasi kriptografi secara lokal — kunci privat tidak pernah meninggalkan perangkat pengguna. Frontend dan backend direncanakan untuk di-deploy secara terpisah pada platform Vercel. Arsitektur sistem secara keseluruhan ditunjukkan pada Gambar 3.1.

2.	Iterasi 2: Perancangan Implementasi dan Pengujian Fungsional
Iterasi ini berfokus pada perancangan implementasi fitur utama autentikasi WebAuthn, mekanisme keamanan, fitur layanan administrasi, serta skenario pengujian fungsional, yang meliputi:
a)	Perancangan Backend
Backend dirancang menggunakan Node.js dengan Express dan mencakup komponen-komponen berikut.
1)	Konfigurasi Server dan Middleware Keamanan
Server dirancang menggunakan Helmet untuk mengatur HTTP security headers, CORS untuk membatasi akses API dari domain frontend yang diizinkan, dan rate limiting untuk membatasi jumlah permintaan dalam periode waktu tertentu, yaitu 300 permintaan per IP per 15 menit secara global serta 30 percobaan autentikasi per IP per 15 menit pada endpoint autentikasi.

3)	Rancangan Mekanisme Keamanan
Mekanisme keamanan yang dirancang merupakan realisasi dari kebutuhan non-fungsional FN-01 hingga FN-05:
a)	Enkripsi AES-256-CBC: Data kependudukan sensitif dienkripsi menggunakan AES-256-CBC dengan IV acak. Kunci enkripsi disimpan dalam environment variable server.
b)	Blind Index HMAC-SHA256: Email di-hash menggunakan HMAC-SHA256 dan disimpan sebagai atribut emailHash sehingga pencarian dapat dilakukan tanpa mendekripsi seluruh dokumen.
c)	Counter Anti-Replay Attack: Setiap kredensial memiliki counter yang bertambah setiap login. Server menolak respons autentikasi jika counter tidak meningkat.
d)	Recovery Codes dengan Hash SHA-256: Kode pemulihan disimpan dalam bentuk hash SHA-256 dan hanya dapat digunakan satu kali.
e)	JWT: Setelah autentikasi berhasil, server mengeluarkan token JWT dengan masa berlaku 7 hari untuk mengotorisasi akses ke endpoint yang memerlukan autentikasi.
4)	Rancangan Fitur Layanan Administrasi
Sistem dirancang untuk mengimplementasikan fitur pengajuan Surat Keterangan Tidak Mampu (SKTM) sesuai F-06 sebagai media implementasi dan pengujian sistem autentikasi FIDO2/WebAuthn. Setiap pengajuan dirancang untuk menyimpan data pemohon yang terenkripsi dan melalui alur persetujuan admin sebelum dapat diunduh dalam format PDF.
b)	Perancangan Frontend
Frontend dirancang menggunakan React.js dengan Vite dan TailwindCSS, mencakup komponen WebAuthnAuth yang akan memicu prompt biometrik melalui fungsi startRegistration() dan startAuthentication() dari library @simplewebauthn/browser. Halaman yang dirancang meliputi halaman Registrasi, Login, dan Recovery untuk alur autentikasi, halaman Dashboard, Layanan, Form Pengajuan, Riwayat, Manajemen Perangkat, dan Profil untuk warga, serta halaman Dashboard Admin, Manajemen Pengajuan, dan Manajemen Pengguna untuk admin.

3.	Iterasi 3: Penyempurnaan dan Pengujian Keamanan
Iterasi ini berfokus pada penyempurnaan rancangan antarmuka dan perancangan skenario pengujian keamanan, yang meliputi:
a)	Penyempurnaan Rancangan
Penyempurnaan dilakukan pada tiga aspek: penyempurnaan rancangan antarmuka pengguna termasuk penambahan pesan konfirmasi pada setiap tindakan penting, penanganan edge case pada alur autentikasi seperti dukungan counter mismatch untuk autentikator dengan sinkronisasi cloud, serta validasi lintas perangkat untuk memastikan sistem dapat berfungsi pada browser desktop dan mobile.
b)	Pengujian Keamanan
Pengujian keamanan dilakukan melalui tiga skenario untuk membuktikan mekanisme keamanan FIDO2/WebAuthn dalam menghadapi ancaman sesuai rumusan masalah.
1)	Skenario 1: Simulasi Phishing Attack
Tabel 3.10. Skenario Pengujian Phishing Attack
No	Aspek	Deskripsi
1	Tujuan	Membuktikan bahwa autentikasi WebAuthn kebal terhadap serangan phishing.
2	Prosedur	Membuat halaman login palsu pada domain berbeda, meminta pengguna login melalui halaman palsu, dan mengamati respons browser serta server.
3	Mekanisme Pertahanan	WebAuthn menggunakan origin binding — credential hanya dapat digunakan pada domain yang sama dengan domain saat credential didaftarkan. Server memverifikasi origin pengirim terhadap expectedOrigin yang dikonfigurasi.
4	Hasil yang Diharapkan	Browser atau server menolak autentikasi karena origin tidak cocok.

2)	Skenario 2: Simulasi Replay Attack
Tabel 3.11. Skenario Pengujian Replay Attack
No	Aspek	Deskripsi
1	Tujuan	Membuktikan bahwa data autentikasi yang disadap tidak dapat digunakan ulang.
2	Prosedur	Melakukan login sah dan merekam data request ke endpoint /api/auth/webauthn/login/verify, kemudian mengirim ulang data yang sama menggunakan Postman.
3	Mekanisme Pertahanan	(a) Challenge sekali pakai — setelah diverifikasi, challenge dihapus dari server; (b) Counter incrementing — server menolak jika counter tidak meningkat.
4	Hasil yang Diharapkan	Server menolak autentikasi karena challenge tidak valid atau counter tidak meningkat.

3)	Skenario 3: Simulasi Database Breach
Tabel 3.12. Skenario Pengujian Database Breach
No	Aspek	Deskripsi
1	Tujuan	Membuktikan bahwa kebocoran basis data tidak mengekspos data sensitif.
2	Prosedur	Mengakses langsung isi basis data MongoDB dan mengevaluasi apakah data yang tersimpan dapat dipahami atau digunakan penyerang.
3	Mekanisme Pertahanan	(a) Enkripsi AES-256-CBC — data kependudukan tersimpan sebagai ciphertext; (b) Hash satu arah — email dan recovery codes tidak dapat dikembalikan ke bentuk asli; (c) Kunci publik tidak dapat digunakan untuk login tanpa kunci privat.
4	Hasil yang Diharapkan	Data kependudukan tidak terbaca, hash tidak dapat dipulihkan, kunci publik tidak dapat digunakan untuk login.

c)	Evaluasi Akhir
Evaluasi akhir dilakukan dengan menganalisis hasil pengujian keamanan terhadap tujuan penelitian. Evaluasi mencakup verifikasi kesesuaian sistem dengan kebutuhan dan tujuan penelitian, serta pengambilan keputusan kelayakan sistem.
d)	Prototipe Final
Apabila hasil evaluasi akhir menyatakan sistem telah memenuhi seluruh kebutuhan fungsional dan mekanisme keamanan terbukti mampu menghadapi tiga skenario ancaman, maka prototipe dinyatakan final dan siap digunakan. Tahap ini mencakup dokumentasi sistem dan penyusunan hasil akhir pengembangan prototipe.
