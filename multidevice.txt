ANALISIS IMPLEMENTASI MULTI-DEVICE WEBAUTHN (PASSWORDLESS)

1. ALUR KERJA (FLOW) MULTI-DEVICE
Alur kerja untuk mendukung banyak perangkat pada satu akun secara passwordless adalah sebagai berikut:

   a. Registrasi Perangkat Pertama (Initial Registration):
      - User memasukkan email di halaman Sign Up.
      - Server generate registration options (challenge, RP ID, dll).
      - Browser memanggil 'navigator.credentials.create()'.
      - User melakukan autentikasi biometrik/PIN di perangkat.
      - Browser mengirimkan 'attestation object' ke server.
      - Server melakukan verifikasi dan memanggil 'user.addWebAuthnCredential()' untuk menyimpan credential pertama.
      - User diarahkan ke Dashboard.

   b. Menambah Perangkat Baru (Add Device - Session Active):
      - User sudah login di Perangkat A, lalu membuka dashboard di Perangkat A.
      - User memilih menu "Tambah Perangkat" (Settings/Profile).
      - Server memanggil route 'POST /user/credentials/add-options' (memerlukan autentikasi).
      - User menggunakan Perangkat B (misal: Security Key) atau melakukan "Hybrid Registration" (Scan QR dari HP).
      - Browser memanggil 'navigator.credentials.create()'.
      - Server memanggil route 'POST /user/credentials/add-verify' untuk memvalidasi dan menyimpan credential perangkat baru.
      - User kini memiliki 2 perangkat yang terdaftar di database.

   c. Login (Authentication):
      - User memasukkan email di halaman Login.
      - Server mencari User berdasarkan email dan mengambil SEMUA 'credentialID' yang terdaftar.
      - Server mengirimkan list 'allowCredentials' ke browser.
      - Browser menampilkan pilihan perangkat mana yang ingin digunakan (biasanya otomatis memilih yang ada di perangkat tersebut).
      - User autentikasi, browser mengirim 'assertion response'.
      - Server memverifikasi signature dan mencocokkannya dengan public key yang sesuai di database.
      - Login Berhasil.

2. STRUKTUR TABEL (SCHEMA)
Saat ini sistem menggunakan MongoDB dengan Mongoose. Struktur tabel untuk mendukung multi-device diimplementasikan sebagai 'Sub-document Array' di dalam Tabel User.

Daftar field dalam sub-document 'webauthnCredentials':
- credentialID (String): ID unik untuk setiap kunci/perangkat.
- credentialPublicKey (String): Public key yang digunakan untuk verifikasi di server.
- counter (Number): Penghitung tanda tangan (mencegah replay attack).
- deviceType (Enum): "platform" (Fingerprint/FaceID internal) atau "cross-platform" (USB Security Key).
- transports (Array of Strings): Metode komunikasi (USB, NFC, BLE, internal).
- nickname (String): Nama perangkat buatan user (contoh: "Laptop Asus", "HP iPhone").
- createdAt (Date): Tanggal pendaftaran perangkat.
- lastUsed (Date): Terakhir kali perangkat ini digunakan untuk login.

3. STRUKTUR TABEL RECOVERY (WORST-CASE)
Di dalam Tabel User juga terdapat field 'backupCodes':
- code (String - Hashed): Hash SHA-256 dari kode pemulihan.
- used (Boolean): Status apakah kode sudah pernah dipakai.
- createdAt (Date): Tanggal pembuatan kode.

4. STEP-BY-STEP PEMBUATAN MULTI-DEVICE
Untuk membangun sistem ini, langkah-langkahnya adalah:
   Step 1: Definisikan Schema User agar 'webauthnCredentials' berbentuk ARRAY (bukan single object).
   Step 2: Buat Controller 'getRegisterOptions' yang dapat menangani penambahan credential ke User yang sudah ada (berdasarkan email atau session).
   Step 3: Pastikan Controller 'getLoginOptions' mengirimkan SEMUA 'credentialID' di array 'webauthnCredentials' user tersebut ke dalam property 'allowCredentials'.
   Step 4: Buat UI di halaman Profile/Settings yang menampilkan daftar perangkat terdaftar, fitur untuk menghapus perangkat (Revoke Account), dan fitur "Add New Device".
   Step 5: Implementasikan fitur "Recovery Codes" saat pendaftaran pertama agar user memiliki akses cadangan jika semua perangkat hilang.

6. CARA MENGATASI WORST-CASE SCENARIO

   Skenario A: Kehilangan Seluruh Perangkat (Misal: HP Hilang & Laptop Rusak bersamaan)
   Solusi: Menggunakan "Backup Codes" (Recovery Codes).
   - User memasukkan email di halaman Recovery.
   - User memasukkan salah satu dari 8-10 Recovery Code yang diunduh saat registrasi awal.
   - Server memverifikasi hash kode tersebut. Jika valid, user diberi session sementara (temporary token).
   - User HARUS segera mendaftarkan perangkat baru di sesi tersebut.
   - Kode pemulihan yang sudah dipakai akan ditandai 'used: true'.

   Skenario B: Perangkat Dicuri/Hilang (Compromised Device)
   Solusi: "Remote Revocation".
   - User login menggunakan perangkat lain yang masih aktif.
   - Masuk ke menu "Security" atau "Device Management".
   - Klik "Hapus" pada perangkat yang hilang.
   - Server menghapus credentialID perangkat tersebut dari database.
   - Pencuri tidak bisa lagi login menggunakan perangkat tersebut karena public key sudah dihapus.

   Skenario C: Counter Mismatch (Deteksi Kloning Perangkat)
   Solusi: Keamanan Protokol WebAuthn.
   - Server selalu mengecek apakah 'counter' dari browser lebih besar daripada yang ada di database.
   - Jika 'counter' lebih kecil atau sama, ada kemungkinan perangkat dikloning atau diserang replay attack.
   - Server akan menolak login dan mengirim notifikasi keamanan.
